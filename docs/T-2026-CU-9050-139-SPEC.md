# T-2026-CU-9050-139 — Bypass-Reader-Rewiring + reversibler Write-Path-Cutover (C-Gate Phase 5 prep)

## Intent

C-Gate Phase 5 (Drop der ~9,3k Per-Coin-`{SYM}_{tf}[_indicators]`-Tabellen) ist blockiert,
solange noch **laufender** Code diese Tabellen per Raw-SQL liest. Dieser Task stellt jeden
aktiven Direct-per-Coin-Reader auf `core.candles` um (hyper-fähig seit T-128, live seit
2026-07-16) und baut einen **reversiblen Write-Path-Flag**, mit dem die Writer probeweise
hyper-primär schreiben (Legacy aus), damit Michi den Write-/Storage-Performance-Gewinn
messen kann, **bevor** die Tabellen gedroppt werden.

Zwei-Teile-Task: **(A)** Read-Rewiring aller aktiven Bypass-Sites, **(B)** reversibler
Write-Cutover-Flag.

## Kontext / grounding

- **Read-API bereits vorhanden** (T-034/T-128): `read_candles`, `read_candles_with_indicators`,
  `latest_open_time` — mit `source==hyper` live. Rewiring = Raw-SQL → diese API, byte-parität.
- **Write-API vorhanden** (T-114/T-119): `upsert_candles`/`upsert_indicators` schreiben
  **Legacy primär** + spiegeln hyper wenn `KYTHERA_CANDLES_DUAL_WRITE=1`. Es gibt **keinen**
  hyper-primär-Modus — genau der wird in Teil B gebaut.
- **Muster:** Blocks 1–6 (DESC→ASC via `.iloc[::-1]`, `Decimal`→`float`-Cast, `include_forming`
  Cutoff clock-basiert, keine Consumer-Reindex).
- Decision **D-2026-CLD-109**; harte Regeln CLAUDE.md #5 (nur geschlossene Kerzen), #7 (geteilte
  Feature-Builder = Trainer==Serving), #8 (Caller committet Transaktion).

## Out of Scope (bewusst)

- `legacy_trainers/*` (frozen provenance, `zzz.py` = Scratch — harte Regel: alte Trainer-Provenance
  nicht anfassen) und `db_schema_analysis.py` (Katalog-Diagnose, meldet nach Drop einfach 0).
  **Diese brechen beim Table-Drop — bewusst, dokumentiert.**
- `tools/audit/step7_monitor_replay.py` — TZ-forensic **Wegwerf**-Skript (manuell, CI-exkludiert). Sein
  5m-Gap-Census (`generate_series` über `{coin}_5m`) UND die Coin-Enumeration aus den Per-Coin-Tabellennamen
  hängen strukturell an der Per-Coin-Existenz; voll drop-safe zu machen bräuchte den `SELECT DISTINCT symbol`-
  Scan über die 40M-Zeilen-Hypertable, den T-128 bewusst vermeidet (>20 s). Unverhältnismäßig für ein
  Throwaway → bricht am Drop wie `legacy_trainers`, akzeptiert (Michi-Scope 2026-07-16).
- **Delete-/Retention-Helfer** (`delete_candles_before`, `delete_indicators_from`) honorieren `_write_primary()`
  bewusst **nicht** — AK7 gilt nur für die `upsert_*`-Writes. Unter `WRITE_PRIMARY=hyper` löschen sie weiter
  aus den (dann eingefrorenen) Legacy-Per-Coin-Tabellen; die Hypertable-Retention ist Phase-5-Policy
  (Compression). Benign für den gegateten Write-Only-Perf-Trial. **Operativer Hinweis:** ein
  `tools/recompute_indicators.py`-Lauf während des Trials editierte Legacy (invalidate→recompute), während
  Reads aus hyper kommen — kein read-sichtbarer Effekt, aber Legacy driftet zusätzlich. Delete-Primary-
  Awareness wäre ein Folge-Task, falls der Trial lange läuft.
- Der **tatsächliche** `WRITE_PRIMARY=hyper`-Flip + Fleet-Restart (Michi-gegatet, Perf-Trial).
- Der **Table-Drop** selbst (Phase 5, nach positivem Trial + 7d + pg_dump — Michi-gegatet).
- `11_ai_mis_bot`, `core/research_features` — bereits auf `core.candles` (Blocks 4/5).

## Why Build (Reuse-Verdict Phase 0b)

**Extend**, nicht Build: Read-Sites nutzen die bestehende `core.candles`-Read-API (Reuse);
Teil B **erweitert** `core.candles.upsert_*` um den Primary-Selector. Keine neue Library.

## Risiko-Hinweis

Teil B ist der riskante Teil: ein Bug im Write-Primary-Selector kann dazu führen, dass die
Ingestion **nichts** oder in die **falsche** Tabelle schreibt = Live-Datenverlust. Deshalb:
DB-gated Write-Parität mit Rollback, Default bleibt `legacy`, Flip ist Michi-gegatet.
