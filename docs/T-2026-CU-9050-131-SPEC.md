# SPEC — T-2026-CU-9050-131: Analytics-Export (DuckDB/Parquet) + Erfolgsraten-Endpoint

## Intent
Erster Implementierungs-Task der Z1-Stufe-1 (Kuratierung mcp-a868a761829e). Baut das
Analytics-Datensubstrat des Dashboards: `tools/analytics_export.py` zieht inkrementell per
Watermark nur *geschlossene* Rows aus vier Live-PG-Quellen in ein columnar Substrat (Parquet
datums-partitioniert + eine DuckDB-Datei), plus einen ersten API-Endpoint (Erfolgsraten-Zeitreihe,
Rolling 7/30/90d, Bot-Multiselect), der **ausschließlich gegen DuckDB** liest — nie gegen Live-PG.

## Why Build (Phase 0b Reuse-Check → Build)
- **z-search-first Need:** "inkrementeller PG→DuckDB/Parquet-Sync per Watermark, DB-frei testbar".
- **Verdict:** Build. Der DuckDB/Parquet-Sync ist per Kuratierung (mcp-a868a761829e Punkt 4) der
  *fixierte, einzige* Analytics-Pfad — es gibt kein bestehendes Cu-Tool dafür. Reuse auf
  Bibliotheks-Ebene: DuckDB (native Parquet-Reader/Writer, **kein pyarrow nötig**) statt eigener
  Columnar-Storage. Kein Reinventing — der Code ist reine Orchestrierung um DuckDB.

- [x] **AK1 Inkrementell/Watermark:** Zweiter Export-Lauf ohne neue Rows exportiert 0 Rows; kommen

  `test_incremental_second_run_picks_up_only_new_rows`, `test_same_timestamp_tie_not_skipped_across_runs`.
- [x] **AK2 Nur geschlossene Rows:** Offene/ungefüllte Rows (`close_time IS NULL`,
  `ENTRY_NOT_FILLED`, `posted IS NULL`) werden nicht exportiert. — ✅ `test_open_and_unfilled_rows_excluded`.
- [x] **AK3 Vier Quellen → DuckDB-Tabellen + Parquet:** closed_trades, closed_ai_signals,
  ml_predictions, regime_history landen als DuckDB-Tabellen UND als datums-partitionierte
  Parquet-Dateien (`<src>/dt=YYYY-MM-DD/data.parquet`). — ✅ `test_all_sources_materialise_to_duckdb`,

## Out of Scope (bewusst NICHT gebaut)
- Kerzen-Export (die 25-GB-Quelle; später, nur 5m-Basis-TF).
- Leaderboard/Risiko-Kennzahlen, Regime-Heatmap, Coin-Drilldown, Digest — Folge-Tasks.
- Frontend/Charting (Lightweight Charts / ECharts).
- Echter VPS-Lauf gegen die Live-DB (Build-Maschine hat keine Credentials — nur VPS-Session).
- Framework-Finalisierung (T-2026-CU-9050-130): Endpoint bleibt dünner Flask-Blueprint, zieht später ggf. um.
- Timescale-Umstellung: nur Forward-Kompatibilität (Watermark-Quelle austauschbar via SourceSpec).
